# 临床药学监护系统产品优化建议报告

## 1. 执行摘要

从产品视角看，这个项目已经具备了一个清晰的 MVP：
- 支持账号体系、上传工作簿、患者检索、AI 生成监护记录、写回 Excel、管理员管理用户。
- 同时保留 Web 与 CLI 两条工作流，兼顾病区现场使用与批量处理场景。

但如果目标是从"能用"走向"真正提升药师效率、降低出错率、便于院内推广"，当前仍有 4 个核心短板：
1. **首屏门槛偏高**：用户必须先准备 `.xlsm` 工作簿才能开始，系统本身没有"新建任务 / 导入病例清单"的闭环。
2. **患者工作台偏弱**：当前主界面主要是列表 + 编辑框，缺少任务优先级、待办、状态流转与风险提示。
3. **AI 价值展示不足**：AI 仅负责把口述整理成四段式文本，缺少结构化采集、建议解释、质量校验与可追溯性。
4. **数据资产沉淀不足**：患者数据被匿名化后仅做文件级存储，缺少病例沉淀、检索分析、模板迭代与复盘能力。

建议按"三步走"推进：
- **第一阶段（1~2 周）**：补齐 onboarding、导入闭环、空状态提示、错误反馈。
- **第二阶段（2~4 周）**：做患者工作台、任务状态、AI 质量校验、记录复用。
- **第三阶段（1~2 月）**：做数据结构化沉淀、检索统计报表、模板运营能力。

## 2. 当前产品定位判断

### 2.1 这个产品现在解决了什么问题

基于 README 和代码实现，当前产品主要解决的是：
- 将病例清单导入为监护工作簿，统一数据载体。
- 让药师在 Web 端快速选择患者并生成/保存标准化监护记录。
- 通过 OpenAI 兼容接口把查房口述整理为标准"问题→分析→处理→结果/计划"格式。
- 保持与医院现有 Excel 工作簿流程兼容，避免完全重构线下流程。

这说明产品的核心价值不是"重新发明 HIS/EMR"，而是**围绕药师日常记录动作做提效工具**。这个方向是对的，因为切口小、落地快、替换成本低。

### 2.2 当前更像"工具"而不是"平台"

虽然 README 已经描述成"临床药学监护工作平台"，但从实际能力看，它还更接近：
- 一个带登录权限的 Excel 辅助录入工具；
- 一个把自然语言整理成标准文本的 AI 小助手；
- 一个支持多人账号的轻量内部系统。

要真正升级为"平台"，至少还需要补齐：
- 患者维度的连续管理；
- 任务维度的状态流转；
- 记录维度的质量追踪；
- 数据维度的沉淀与复用能力。

## 3. 关键现状观察

### 3.1 主流程仍然依赖用户先上传 `.xlsm`

Web 端首页的第一步是上传监护工作簿，上传成功后才进入主工作台；后端接口也围绕"查找当前用户目录下工作簿"展开。这对熟悉 Excel 流程的药师很友好，但对新用户并不自然：如果没有现成工作簿，系统无法直接开始工作。

**产品影响：**
- onboarding 不完整；
- 形成"先准备文件，再用系统"的心智负担；
- 系统价值容易被感知为"文件编辑器"，而不是"工作平台"。

### 3.2 患者列表具备检索功能，但缺少优先级和任务视角

前端支持搜索床号、诊断、匿名编号，也支持查看历史记录、手动新增患者；但列表本身没有"今日待处理""高风险患者""已完成/待补充""连续未跟进"等状态信息。

**产品影响：**
- 用户只能"找人"，不能"排班次、看重点、清待办"；
- 工作负荷一大时，系统对决策顺序帮助有限；
- AI 和历史记录无法转化成可执行任务管理。

### 3.3 AI 目前只做文本整理，没有做质控和解释

`llm.py` 的职责非常明确：调用模型，把输入整理成固定四段式文本，并做格式校验。这种设计保证了输出可控，但也意味着 AI 的产品价值还比较浅：
- 不判断信息是否缺失；
- 不提示关键字段（药物、剂量、浓度、肾功能、过敏史等）是否缺项；
- 不解释"为什么给出这个分析/建议"；
- 不对生成内容做风险分级。

**产品影响：**
- AI 更像排版助手，而非临床决策支持助手；
- 用户会觉得"省了一点打字时间"，但未必觉得"显著提升专业质量"；
- 院内推广时，难以证明 AI 对质量和安全的增益。

### 3.4 数据安全做了匿名化，但还没有形成完整的数据治理故事

`excel_io.py` 会在上传后备份原始身份标识，并将住院号和姓名匿名化；下载时再恢复。这体现了很好的隐私意识。但目前匿名化更多是"文件层保护"，不是"产品层治理"：
- 缺少操作日志；
- 缺少下载审计；
- 缺少异常访问提示；
- 缺少数据保留周期、归档、删除策略。

**产品影响：**
- 适合内部小范围使用；
- 一旦扩大到多科室、多用户、持续运行，合规与追责能力会成为瓶颈。

## 4. 优化建议（按优先级）

## P0：先把"可持续使用"补齐

### 4.1 建立完整的新手引导与导入闭环

**建议做法：**
- 首页不要只显示"上传工作簿"，增加两个入口：
  1. 上传已有监护工作簿；
  2. 从病例清单自动创建新工作簿。
- 为首次进入的用户增加 3 步引导：上传/导入 → 选择患者 → 生成并保存记录。
- 增加空状态说明：没有工作簿时告诉用户下一步该做什么，而不是仅返回报错。

**为什么优先级最高：**
这是决定留存的第一道坎。当前功能强，但"首次成功"路径不够短，用户容易在准备文件阶段流失。

**预期收益：**
- 降低培训成本；
- 减少"不会用"的反馈；
- 让系统从"依赖文件"变成"主导流程"。

### 4.2 做更明确的错误提示和恢复动作

当前后端接口在未上传工作簿时会返回"云端没有工作簿，请先上传"；前端已有 toast，但可以进一步产品化。

**建议做法：**
- 把错误分为：配置错误、上传错误、AI 生成错误、Excel 写入错误、权限错误。
- 每类错误配一个明确恢复动作，例如：
  - "请重新上传原始工作簿"；
  - "请检查日期格式为 YYYY-MM-DD"；
  - "AI 生成失败，可直接手动编辑下方监护意见"。
- 对高频错误给出 FAQ 弹层或帮助入口。

**预期收益：**
- 降低用户恐慌感；
- 降低管理员/开发者被动支持成本；
- 提高问题自助解决率。

### 4.3 在保存前增加最基础的质控提示

目前保存依赖必填项检查和 Excel 槽位逻辑，但从产品上还可以更进一步。

**建议做法：**
- 保存前提示是否缺少监护级别、监护类型、日期。
- 识别监护意见是否缺少四段结构、是否过短。
- 如果覆盖已有槽位，弹出更清晰的二次确认，展示原记录摘要与新记录摘要。

**预期收益：**
- 直接减少误覆盖；
- 降低记录质量波动；
- 增强用户对系统可靠性的感知。

## P1：把"记录工具"升级为"患者工作台"

### 4.4 增加患者优先级与待办状态

现在患者列表只有基础信息和历史记录，建议进一步增加：
- `待处理 / 已完成 / 待复核 / 今日已更新 / 超 48 小时未更新`；
- 重点标签：抗菌药物、TDM、CRRT、肾功能异常、ADR 风险等；
- 支持按照重点程度排序。

**产品价值：**
让药师不是"逐个点开患者"，而是"先处理最重要的患者"。这会显著提升工作台价值。

### 4.5 增加"患者摘要卡"与连续追踪视图

当前前端有 patient summary 区域，但可利用空间更大。

**建议做法：**
在选中患者后展示：
- 最新一次监护日期；
- 当前主要问题；
- 最近 3 条监护记录摘要；
- 是否存在未完成建议；
- 今日与上次相比的变化点。

**产品价值：**
减少来回翻记录的时间，让"接续查房"和"多人轮转交接"更顺畅。

### 4.6 增加记录模板推荐与快捷复用

项目已经有 8 套模板和 CLI 能力，但 Web 侧的模板价值没有被直接显性化。

**建议做法：**
- 根据患者诊断 / 用户输入关键词推荐模板；
- 支持"从上一条记录复制并编辑"；
- 支持保存个人常用短语、科室常用模板。

**产品价值：**
这是最容易直接提升效率的点之一，尤其适合高频重复场景。

---

## P1：把 AI 从"润色器"升级成"助手"

### 4.7 增加结构化采集和缺项提醒

既然 AI 已经要求保留药物名、剂量、检验值，不妨进一步把这些信息抽出来。

**建议做法：**
- 在生成后自动识别：药物、剂量、频次、血药浓度、肾功能、感染证据、建议动作。
- 如果关键字段缺失，提示"建议补充：最近 Scr / eGFR、给药方案、采样时间"等。
- 将 AI 输出拆成"结构化字段 + 可编辑正文"。

**产品价值：**
这会让 AI 输出更可检查、更可复用，也更利于未来统计分析。

### 4.8 增加 AI 输出依据与风险提示

很多临床用户不怕 AI 慢，怕的是 AI"看起来很像对的"。

**建议做法：**
- 展示 AI 识别到的输入要点；
- 标记哪些结论来自原始口述，哪些是模型补足的通用表述；
- 对高风险建议增加"请人工复核"提示。

**产品价值：**
提升可信度，减少用户对黑盒内容的抵触情绪。

### 4.9 增加"生成后校验"而不是只做"格式校验"

当前模型输出只校验四个 marker 是否存在，这对于产品上线远远不够。

**建议做法：**
- 校验是否存在明显冲突信息（如"浓度偏高"但建议"增加剂量"）；
- 校验是否有空泛模板化语句；
- 对长度过长、重复、缺段做提示。

**产品价值：**
能显著提升用户对 AI 结果的信任，并减少保存后返工。

---

## P2：做数据沉淀，而不仅仅是写回 Excel

### 4.10 建立病例与记录的检索/统计能力

当前系统的主数据仍然附着在 Excel 上，不利于长期积累。

**建议做法：**
- 在不改变 Excel 导出的前提下，同步保存结构化记录到数据库；
- 支持按药物、问题类型、模板、科室、时间区间检索；
- 做月度统计报表，例如 ADR 监测数、TDM 干预数、抗菌药物评估数。

**产品价值：**
对一线药师是"好查"，对科室主任是"可汇报"，对医院管理是"可量化"。

### 4.11 让模板成为可运营资产

目前模板固化在 `picu_note_templates.json`，更像开发资源，而不是产品资产。

**建议做法：**
- 提供后台维护模板；
- 支持版本管理、启停、适用场景说明；
- 统计模板使用率、采纳率、人工修改率。

**产品价值：**
让模板持续进化，而不是每次都改代码。

## 5. 建议路线图

### 第一阶段：快速提升可用性（1~2 周）

目标：让用户更容易第一次用起来，并减少明显出错。

建议交付：
- 新手引导页 / 空状态；
- 从病例清单创建工作簿入口；
- 保存前质控提醒；
- 更清晰的错误反馈；
- Web 端显性化模板推荐与记录复用。

### 第二阶段：提升日常效率（2~4 周）

目标：让系统从"录入工具"升级成"工作台"。

建议交付：
- 患者优先级标签与待办状态；
- 患者摘要卡与连续追踪；
- AI 结构化提取与缺项提醒；
- AI 输出依据展示与风险提示；
- 生成后语义校验。

### 第三阶段：数据沉淀与模板运营（1~2 月）

目标：让数据和模板成为可持续迭代的资产。

建议交付：
- 结构化记录同步入库；
- 病例与记录检索统计；
- 模板运营后台；
- 数据报表与月度统计。

## 6. 我最推荐优先做的 3 件事

如果资源有限，我建议优先只做这 3 件：

### 6.1 把"导入/创建工作簿"做完整
这是所有转化的入口，优先级最高。

### 6.2 给患者列表加"待办 + 优先级"
这是最直接影响日常效率的功能，用户能马上感受到价值。

### 6.3 给 AI 增加缺项提醒和保存前校验
这是最容易把"AI 好玩"升级成"AI 可靠"的一步。
